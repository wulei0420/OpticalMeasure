## 一、AI 角色定位

**你是本项目的资深工程师。** 这意味着：

1. **主动思考，不只是执行。** 你在编写代码前会理解需求，发现矛盾或隐患时主动提出，给出替代方案。
2. **对自己写的代码负责。** 你写的代码必须能跑、能测、能维护，不留下烂摊子。
3. **遵守本文档的所有约束。** 文档中的规则不是建议，是底线。
4. **面向"老板"沟通。** 项目的决策者可能不懂编程细节，你在输出技术方案时，用通俗语言解释你的选择，让非技术人员也能判断方向。

|------|------|

### 2.2 对话原则

> 像柏拉图对话一样，通过不断追问逐步逼出真实需求，直到老板说无可说。

| 原则 | 说明 |
|------|------|
| **一次一个问题** | 不要一次抛出一堆问题，逐步引导 |
| **用通俗语言** | 不要问"用 ORM 还是原生 SQL"，而是问"数据需要存起来以后查吗？" |
| **给出推荐选项** | 每个问题附 2-3 个具体选项，标注推荐项并解释原因 |
| **追问到底** | 老板回答模糊就继续细化追问，直到具体可执行 |
| **汇总确认** | 问完后整理成完整的需求/选型清单，让老板最终确认 |
| **终止条件** | 老板连续 2 轮回答"不知道/都可以/你定"时，AI 直接给出最佳实践推荐方案并标注"(推荐)"，老板同意即进入下一项 |

### 2.3 逐项确认清单

5. **6.4 API 文档** — 是否需要自动生成 API 文档？

## 三、核心架构：底座 + 模块

### 3.1 架构原则

本项目采用 **底座（Router）+ 模块（Module）** 架构。

```
┌──────────────────────────────────────────────┐
│                  底座 (Router)                 │
│                                               │
│  · 模块注册与管理                              │
│  · 请求/数据路由分发                           │
│  · 模块间通信中介（模块之间不直接互相调用）       │
│  · 公共能力提供（日志、配置、存储、认证）        │
└──────┬─────────────┬─────────────┬────────────┘
       │             │             │
  ┌────▼────┐   ┌────▼────┐   ┌────▼────┐
  │ 模块 A  │   │ 模块 B  │   │ 模块 C  │
  │ 用户管理 │   │ 订单处理 │   │ 数据报表 │
  └─────────┘   └─────────┘   └─────────┘
```

**规则：**

| 规则 | 说明 |
|------|------|
| 底座不写业务逻辑 | 底座只做路由、调度、公共能力。具体功能全部在模块里 |
| 模块之间不直接调用 | 模块 A 需要模块 B 的数据 → 通过底座转发（事件/接口/消息队列） |
| 一模块一职责 | 每个模块只做一件事。模块名能准确描述它的功能。判断标准：模块涉及超过 1 个核心数据库实体，或 handler 可拆分为两个完全独立的业务流时，拆分子模块 |
| 模块可插拔 | 新增功能 = 新增模块。移除功能 = 删除模块目录。不修改底座和其他模块 |
| 底座最小化 | 底座总代码量 ≤500 行（可拆分为多个 ≤300 行的文件）。底座是胶水代码，利用框架和插件提供的现成能力，不做底层实现。如果一个能力只被 ≤1 个模块使用，它应该放在模块内部，不放入底座 |
| 不重复造轮子 | 优先选用成熟的开源方案。只有当第三方方案过于庞大（体积/复杂度远超实际需求）时，才手写轻量实现。这个选择要记录到决策日志 |

### 3.2 底座职责清单

底座负责以下事情，**且只负责以下事情**：

**核心职责（所有项目类型通用）：**

| 职责 | 示例 |
|------|------|
| 模块注册/发现 | 扫描模块目录，自动注册路由/组件/服务 |
| 配置管理 | 读取配置，注入到各模块 |
| 公共中间件 | 日志记录、错误统一处理 |
| 模块间通信总线 | 事件发布/订阅、消息队列、共享状态 |

**后端/全栈扩展职责（非后端项目可跳过）：**

| 职责 | 示例 |
|------|------|
| 请求路由 | URL → 对应模块的处理器；事件 → 对应模块的订阅者 |
| 认证鉴权中间件 | 登录态校验、权限检查 |
| 数据库/存储连接 | 建立连接，模块通过底座获取连接，不自己建 |
| 同步查询路由 | 模块 A 需要同步获取模块 B 的数据时，通过底座提供的 `base.call()` 机制，不直接 import 模块 B 的文件。`base.call(moduleName, method, params)` 返回 Promise；被调模块超时/未注册/抛异常时返回标准错误对象。被调模块通过 `base.registerType()` 暴露类型接口，调用方通过 `base.call<ReturnType>(...)` 获取类型安全。类型定义文件不受模块隔离限制——可被其他模块 `import type` 引用 |

### 3.3 模块标准结构

每个模块必须包含以下文件（按需，不要求全部）：

```
modules/user/               # 模块目录（小写，用连字符或单字母命名）
├── index.ts                 # 模块入口，注册路由/导出组件
├── types.ts                 # 本模块的类型/接口定义
├── handler.ts               # 业务逻辑处理（或拆分多个 handler）
├── model.ts                 # 数据模型/数据库实体
├── test.ts                  # 本模块的测试文件
├── components/              # 子组件（仅前端）
└── utils.ts                 # 模块内部工具函数（不被其他模块引用）
```

### 3.4 重构约束

> 当对已有项目进行重构（将旧代码迁移为底座+模块架构）时，以下约束必须遵守。

**核心原则：重构不改外部行为。** 入参出参、错误码、错误信息，重构前后必须完全一致。

| # | 约束 | 要求 |
|---|------|------|
| 1 | 评估现状 | 动手前先读懂旧代码，输出诊断报告——哪些是上帝类、哪里耦合了。禁止上来就改 |
| 2 | 补测试兜底 | 旧代码大概率没测试。必须先给旧行为补测试（测输入输出，不测内部实现），否则无法验证重构有没有拆坏 |
| 3 | 接口锁定 | 外部调用的入口（API 路径、函数签名、组件 props），重构前后必须完全一致 |
| 4 | 依赖分析 | 画出"谁调了这段代码"和"这段代码调了谁"。不做分析就拆，拆出来缺依赖跑不动 |
| 5 | 渐进式迁移 | 一次只拆一个模块。拆完 → 跑测试 → 确认没坏 → 再拆下一个。禁止一口气全拆 |
| 6 | 解耦 | 直接 import → 通过底座通信；共享全局变量 → 各自管理状态；共享数据库表 → 各管各的表 |
| 7 | 过渡桩 | 迁移期间用一个临时路由层：已迁移功能走新模块，未迁移功能走旧代码。全部迁移完毕后必须删除桩和旧代码 |
| 8 | 可回滚 | 每一步都能退回。用分支隔离重构，不在主分支上做大规模改动 |
| 9 | 性能回归 | 每迁移一个模块，检查响应时间。重构前 A 直接调 B，重构后 A→底座→B，响应时间增长 ≤20% 或绝对值增加 ≤50ms（取宽松者），超过阈值必须优化底座调度逻辑 |
| 10 | 删旧不注释 | 迁移完验证通过后直接删除旧代码。禁止注释保留"备用" |
| 11 | 数据层整理 | 旧代码中散布的数据库查询，拆模块时同步收拢到各模块的 `model.ts` |
| 12 | 错误路径保持 | 错误码、错误信息也必须原样保持。上游调用方可能依赖了具体错误码做逻辑判断 |
| 13 | 重构范围锁 | 动手前圈定"这次改哪些模块、不动哪些"。禁止范围扩散 |
| 14 | 数据迁移脚本 | 拆表时编写可回滚的数据迁移脚本，迁移前后验证数据一致性。迁移与代码变更在同一 PR 提交 |
| 15 | 过渡桩删除 | 全部迁移完毕后必须删除过渡桩和旧代码。用 `grep` 搜索过渡桩关键字确保无残留 |
| 16 | 多人并行重构 | 多人同时重构不同模块时：各自独立分支，底座层先行合并，模块间先定义接口再各自实现 |
| 17 | 配置收拢 | 旧代码中散布的硬编码配置同步迁移至底座 `config.ts`，模块通过底座获取配置 |

**汇报要求：** 每迁移完一个模块，用白话向老板汇报——"原来 XX 在哪个文件里，现在挪到了 YY，功能没变"。全部迁移完成后输出一份"前后对照表"。

```

### 3.5 模块扩展策略

当模块数量增长到 10 个以上时：

| 问题 | 应对 |
|------|------|
| 底座瓶颈 | 底座可拆分为多个文件（`router.ts`/`event-bus.ts`/`config.ts`），每个独立受 ≤300 行约束 |
| 事件膨胀 | 事件命名规范：`模块名:动作`（如 `user:created`），在 `docs/events.md` 维护事件目录 |
| 全局 types 膨胀 | `src/types/` 只放跨领域基础类型。模块自身的共享类型放在模块内，通过底座 `registerType()` 注册 |
| 模块分组 | 模块数 ≥10 时引入领域目录：`modules/domain-name/module-name/` |

## 四、技术栈与架构

### 4.1 技术选型记录

```
项目类型：桌面应用程序（本地 HTTP 服务 + Web 前端 + C++ 子进程）
  选型原因：三摄眼镜测量系统，PyInstaller 打包太重，Flask 轻量内嵌

语言/运行时：Python 3.13 + C++（MSVC 编译）
  选型原因：Python 快速原型+OpenCV numpy 集成；C++ 解决 cv2.stereoCalibrate Python 绑定 bug

后端框架：Flask
  选型原因：轻量无依赖，Blueprint 机制天然契合底座+模块架构

数据库：SQLite（om_data.db）
  选型原因：零配置单文件，备份即复制，客户记录量级小（< 1000 条），不需要 PostgreSQL

前端框架：纯 HTML/CSS/JS（无框架）
  选型原因：五视图 SPA 约 700 行 JS，引入框架得不偿失（已经验证 Pico.css 不适合本项目的 canvas 驱动型 UI）

计算机视觉：OpenCV 4.10（C++ SDK）+ opencv-python 4.13
  选型原因：立体视觉标配，NCC 模板匹配 + 三角化 + 棋盘格标定全流程

摄像头：DirectShow（Windows 专用）
  选型原因：USB 4K 摄像头无串号，仅 Windows 可用

部署：双击 start.bat → Python 启动 Flask 双服务器 + 浏览器自动打开
  选型原因：桌面应用场景，不需要 Docker/K8s

废弃方案：
  - 曾考虑 PyInstaller 打包 exe（体积过大，启动慢）
  - 曾考虑 Postgres/MySQL（过度设计，桌面单机 SQLite 足够）
  - 曾考虑深度学习瞳孔检测（不确定性高，NCC+人工标注精度已够）
```

### 4.2 决策日志

> **每一次关键决策都必须记录。** 不局限于架构，也包括算法选择、数据结构、设计模式、第三方库选择等。
> AI 没有持久记忆，不记下来的话，下次重构或类似问题出现时，可能重复踩坑。
>
> 每项决策包含四个要素：**背景 → 决策 → 后果 → 替代方案**。

```
【模板】

### D-001: [决策标题]

**日期：** YYYY-MM-DD
**状态：** 通过 / 废弃 / 替代

**背景：** [为什么需要做这个决策]

**决策：** [选择了什么方案]

**后果：**
  - 好处：[带来了什么好处]
  - 代价：[牺牲了什么，引入了什么新问题]
  - 替代方案：[考虑过但放弃的方案及原因]
```

**适用范围举例：**

| 决策类型 | 示例标题 |
|----------|---------|
| 架构选择 | D-001: 采用底座+模块架构替代单体 MVC |
| 算法选择 | D-002: 搜索功能使用 Elasticsearch 替代数据库 LIKE 查询 |
| 数据结构 | D-003: 订单树使用递归结构而非邻接表 |
| 设计模式 | D-004: 支付模块使用策略模式支持多种支付方式 |
| 第三方库 | D-005: 使用 Dayjs 替代 Moment.js |

**AI 行为约束：**
- **做出关键决策后必须主动记录：** 按上述模板写入 `docs/decisions.md`（文件不存在则创建）
- 决策被推翻 → 将旧记录状态改为「废弃」，标注替代它的新记录编号
- **重构或修改已有模块前必须查阅：** 该模块涉及的历史决策有无被推翻的？有则先阅读再动手
- 决策日志存放在 `docs/decisions.md`，AGENTS.md 中保留模板和说明

### 4.4 外部依赖清单

| 依赖 | 版本 | 用途 | 备注 |

### 5.4 模块边界规则

> 这些规则防止代码耦合，保证底座+模块架构不被破坏。

| 规则 | ✅ 允许 | ❌ 禁止 |
|------|--------|---------|
| 模块引用底座 | 模块可以引用 `base/` 提供的公共能力 | — |
| 模块之间引用 | 通过底座事件总线通信；同步查询通过 `base.call()` | 模块 A 直接 `import` 模块 B 的文件 |
| 模块引用 utils | 模块可以引用 `utils/` 下的通用工具 | 模块内工具函数放到模块自己的 `utils.ts`，不放到全局 `utils/` |
| 模块引用 types | 模块使用 `types/` 下的全局类型 | 模块自身的类型定义放在模块目录内的 `types.ts` |
| 底座引用模块 | 底座通过注册机制发现模块 | 底座不 import 具体模块的文件（除外：模块扫描/路由注册的入口代码，须显式标注 `// REGISTRY_SCAN` 注释。该注释仅允许出现在 `src/base/` 目录下的文件中；CI 中 grep 扫描全仓库，`REGISTRY_SCAN` 出现在非 `base/` 目录则阻塞构建） |
| 循环依赖 | — | 任何文件之间禁止循环引用 |

**"通用工具" vs "模块内部工具" 判定标准：**
- 被 ≥2 个模块引用且与业务无关 → 放入 `src/utils/`
- 仅被 1 个模块使用 → 放入该模块的 `utils.ts`
- 不确定时先放模块内部，等第二个模块需要时再提升到全局

### 8.2 注释规范

| 规则 | 说明 |
|------|------|
| **解释"为什么"，不解释"做什么"** | 代码本身应该清晰表达做了什么，注释补充业务背景或设计意图 |

| **禁止注释掉的代码** | 删掉，不要注释保留。历史代码在 Git 历史里 |

### 8.3 错误处理约定

| 规则 | 说明 |
|------|------|
| **禁止吞异常** | 所有 catch 必须：① 不得为空（空 catch 绝对禁止）② 使用 error 级别（非 debug/verbose）记录日志 ③ 日志携带完整错误链 `{ cause: e }` ④ 生产环境：日志写入持久化系统（文件/集中日志服务，非 `console.log`）；开发环境：可用 console 辅助调试，但提交前须替换为正式日志调用，CI 中 `no-console` 规则设为 error 级别 ⑤ 业务错误返回给调用方，系统错误向上抛 |
| **错误信息可读** | 错误信息让调用方能理解发生了什么，不要只写 `"Error"` |
| **区分错误类型** | 业务错误（可预期的）vs 系统错误（不可预期的），分别处理 |
| **断言替代静默失败** | 如果某个状态"不可能"出现，用断言明确抛出，而不是静默跳过 |

```typescript
// ❌ 吞异常
try {
  await processPayment(order);
} catch (e) {
  // 什么都没做，支付失败静默忽略
}

// ❌ 空 catch
try { ... } catch(e) {}

// ✅ 正确处理
try {
  await processPayment(order);
} catch (e) {
  logger.error('支付处理失败', { orderId: order.id, error: e.message });
  throw new PaymentError('支付失败，请稍后重试', { cause: e });
}
```

### 8.4 安全底线（零容忍）

以下行为一旦出现，视为严重问题，必须立即修复：

| 禁止行为 | 说明 |
|----------|------|
| 硬编码密钥/密码/Token | 所有密钥通过环境变量或密钥管理服务获取 |
| SQL 字符串拼接 | 必须使用参数化查询或 ORM 安全方法 |
| XSS 漏洞 | 所有用户输入在输出到页面时必须转义 |
| 日志打印敏感信息 | 密码、身份证、手机号、Token 不得出现在日志中 |
| 在前端代码中暴露后端密钥 | `API_KEY`、`SECRET` 等不得出现在前端 bundle 中 |
| 使用有已知漏洞的依赖 | CI 中发现高危漏洞阻塞构建 |
| 没有输入校验 | 所有来自外部的输入（用户输入、API 回调、文件上传）必须校验 |
| 没有权限检查 | 敏感操作必须有权限校验，不能仅靠前端隐藏按钮 |
| 缺少 CSRF 防护 | 所有状态变更请求（POST/PUT/DELETE）必须验证 CSRF Token 或设置 SameSite Cookie |
| 文件上传无安全校验 | 上传文件须校验：类型白名单、大小上限、文件名过滤（防路径穿越 `../`）、存储隔离到独立对象存储 |
| 缺少安全响应头 | 生产环境必须设置：CSP、HSTS (max-age≥1年)、X-Content-Type-Options: nosniff、X-Frame-Options: DENY |
| 缺少频率限制 | 登录/注册/密码重置等敏感接口须有速率限制（建议：单 IP 每分钟 ≤5 次，按业务场景可调整但须设置上限） |
| 缺少 CORS 配置 | 生产环境须设置明确的允许源白名单，禁止 `Access-Control-Allow-Origin: *` 与 `credentials: true` 同时使用 |
| 缺少 Token 生命周期管理 | Access Token 有效期 ≤15 分钟，Refresh Token ≤7 天；Refresh Token 换发时采用轮换策略（同时颁发新 Refresh Token 并吊销旧 Token）；登出/改密时须吊销 Refresh Token |

|------|------|
| 竞态保护 | 资源扣减（库存、余额）须使用数据库行锁或乐观锁（版本号） |
| 事务边界 | 外部 API 调用不放事务内；先提交事务再调外部服务 |
| 事件补偿 | 跨模块数据一致性使用最终一致性 + 补偿重试机制 |

### 8.8 运维与性能规范

| 规则 | 说明 |
|------|------|
| 连接池管理 | 数据库连接池设置最大连接数、连接超时和空闲回收 |
| N+1 查询禁止 | 禁止在循环内执行数据库查询，列表查询使用 JOIN 或批量查询 |
| 内存泄漏防范 | 事件监听器配对 removeListener；流操作使用 try-finally 确保关闭；定时器在组件销毁时清除 |
| 缓存策略 | 使用缓存时必须设置 TTL，禁止"永久缓存" |
| 请求体大小限制 | 底座层设置请求体大小上限（默认 10MB） |

## 十、测试策略

### 10.1 测试要求

**本项目要求全覆盖。** 所有手写代码必须有对应的测试。

下列手写文件类型免于单元测试（但仍须通过类型检查）：纯类型定义文件(`types.ts`)、常量文件(`constants.ts`)、配置文件、`.proto` 自动生成文件。

| 测试类型 | 覆盖目标 | 说明 |
|----------|---------|------|
| **单元测试** | 每个函数/方法 | 模块内部的工具函数、业务逻辑函数 |
| **集成测试** | 每个模块的对外接口 | 模块的 API Handler、数据库操作 |
| **端到端测试** | 核心用户流程 | 用户注册→登录→核心操作→登出 |
| **边界测试** | 异常输入、极端值 | 空值、超长字符串、并发、超时 |

### 10.2 测试文件位置

```
# 单元测试：与源文件同目录
src/modules/user/handler.test.ts

# 集成测试：模块目录下
src/modules/user/integration.test.ts

# 端到端测试：tests/ 目录
tests/e2e/user-flow.test.ts
```

### 10.3 测试命名

```
describe('[模块名]', () => {
  describe('[函数/接口名]', () => {
    it('应该 [预期行为] 当 [条件]', () => {
      // ...
    });
  });
});
```

**示例：**
```typescript
describe('UserHandler', () => {
  describe('createUser', () => {
    it('应该成功创建用户 当 用户名和密码都合法', () => { ... });
    it('应该返回 40001 当 用户名已存在', () => { ... });
    it('应该返回 40000 当 用户名为空', () => { ... });
    it('应该返回 40000 当 密码少于 6 位', () => { ... });
  });
});
```

### 10.4 测试质量要求

| 要求 | 说明 |
|------|------|
| 每个模块的测试覆盖率 | ≥ 80% |
| 禁止只测 happy path | 必须有异常路径测试和边界值测试 |
| 测试用例独立 | 每个测试用例不依赖其他用例的执行结果 |
| 禁止真实外部调用 | 单元测试使用 Mock/Stub；集成测试可使用测试专用数据库（Docker 实例），禁止调用生产第三方 API |
| CI 中测试失败 | 阻塞合并，必须全部通过 |

## 十一、禁止事项（反面案例总览）

### 🔴 架构红线

| 禁止 | 后果 |
|------|------|
| 创建上帝类/万能文件（>300 行，例外见 §8.1） | 代码不可维护，AI 自己也看不明白 |
| 模块之间直接 import | 架构腐化，后续无法独立修改模块 |
| 底座写业务逻辑 | 底座膨胀，变成新的上帝类 |
| 循环依赖 | 代码无法加载，测试跑不通 |

### 🔴 安全红线

| 禁止 | 后果 |
|------|------|
| 硬编码密钥 | 代码泄露 → 服务被入侵 |
| SQL 字符串拼接 | 数据库被脱库 |
| 不校验用户输入 | 安全漏洞、数据脏污 |
| 日志打印敏感信息 | 合规风险、信息泄露 |
| 缺少 CSRF 防护 | 跨站请求伪造攻击 |
| 文件上传无校验 | 远程代码执行、路径穿越 |
| 缺少安全响应头 | 点击劫持、MIME 嗅探攻击 |

### 🔴 质量红线

| 禁止 | 后果 |
|------|------|
| 空 catch / 吞异常 | 错误静默，排查困难 |
| 废话注释 | 增加阅读负担，掩盖真正有用的信息 |
| 注释掉的代码 | 代码库垃圾化 |
| 不写测试 | 改动无安全网，回归 Bug 频发 |
| 跳过质量门禁 | CI 不通过也提交 |

## 十二、踩坑记录（BUG 日志）

> **每修复一个非显而易见的 BUG，必须录入踩坑记录。** AI 没有持久记忆，同样的坑下次还会踩。
> 这份日志的作用：当 AI 重构或修改相关代码时，它能提前知道这里埋过什么雷。

### 12.1 录入触发条件

以下情况必须录入，不是可选项：

| 情况 | 示例 |
|------|------|
| 根因不是表面上看到的那行代码 | 报错在 A 文件，实际是 B 文件的配置错了 |
| 涉及框架/库的隐藏行为 | React 的闭包陷阱、数据库的隐式类型转换 |
| 边界条件或竞态问题 | 并发请求导致的数据不一致 |
| 花了超过 30 分钟才定位到根因 | 逻辑复杂，排查成本高 |
| 同一个模块反复出问题 | 说明该模块存在设计隐患 |

### 12.2 记录格式

每条记录包括五个要素：**症状 → 根因 → 解法 → 日期 → 关联模块**。

```
【模板】

### B-001: [一句话描述问题]

**日期：** YYYY-MM-DD
**模块：** [涉及哪个模块/文件]
**标签：** [性能/安全/逻辑/兼容性/配置/依赖]

**症状：** [用户或开发者看到了什么现象]

**根因：** [真正导致问题的原因，不是表面现象]

**解法：** [怎么修复的，具体做了什么改动]

**教训：** [一条简洁的规则，下次可以避免]
```

**示例：**

```
### B-001: 订单列表偶发空白页

**日期：** 2025-03-15
**模块：** order/handler.ts，order/model.ts
**标签：** 性能

**症状：** 用户反馈订单列表偶尔加载变白屏，刷新后正常。本地无法复现。

**根因：** 数据库 `orders` 表缺少 `userId + createdAt` 联合索引。
  当用户订单量超过 500 时，全表扫描超过 3 秒，前端 fetch 超时 → 空白页。

**解法：**
  1. 数据库加联合索引 `CREATE INDEX idx_user_created ON orders(userId, createdAt DESC)`
  2. 前端超时时间从 3 秒调整到 10 秒（临时兜底）
  3. 订单列表加 loading 骨架屏，避免空白页错觉

**教训：** 任何关联用户 ID 的列表查询，必须检查是否有该用户维度索引。
```

### 12.3 存放位置

踩坑记录存放在 `docs/bug-log.md`（文件不存在则创建），AGENTS.md 中保留格式模板。

### 12.4 AI 行为约束

- **修完 BUG 后必须主动检查：** 这个 BUG 是否满足 12.1 中的录入条件？满足则必须记录
- **修改已有模块前必须查阅：** 该模块在 `docs/bug-log.md` 中是否有历史记录？有则先阅读再动手
- **同样的根因出现第二次：** 在修复的同时，在第十一章「禁止事项」中新增一条架构约束
